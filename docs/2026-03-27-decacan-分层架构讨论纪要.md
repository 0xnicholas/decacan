## 一、总体定位

`decacan` 不是通用聊天助手，也不是 coding agent。

它的核心产品承诺是：

`把一个工作目录和一种做事方式交给我，我尽量少打扰你，并直接产出结果。`

围绕这个承诺，前面的讨论收敛出几个关键原则：

- 产品以 `Playbook` 为入口对象，而不是聊天框
- 产品以 `Artifact` 为结果对象，而不是回答文本
- `标准模式` 是 MVP 主线，强调稳定自动产出
- `发现模式` 提供探索能力，但不应反客为主
- `Psi` 只是被 runtime 在局部 step 中调用的内部智能执行内核，不是产品主语
- 所有真实动作都应走统一受控边界

## 二、总分层视图

当前最稳的分层主线是：

```text
decacan-app
  -> decacan-runtime
       -> psi
```

但在内部理解上，需要继续拆成更细的层：

```text
1. 体验层（Experience Layer）
2. 应用接口层（Application Layer）
3. 产品编排层 / 控制面（Product Domain / Control Plane）
4. 执行协调层 / 数据面（Execution Coordination / Data Plane）
5. Psi 内核层（Psi Kernel）
6. 基础设施层（Infrastructure）
```

同时，还可以从对象视角再收成三组：

- 用户对象
- 运行对象
- 执行对象

## 三、第一层：体验层

这一层是用户直接看见和操作的产品表面。

主要对象和页面包括：

- `Workspace Home`
- `Playbook Catalog`
- `Task Composer`
- `Task Detail`
- `Approval Surface`
- `Artifact Viewer`

这一层承担的职责是：

- 选择工作目录
- 选择 `Playbook`
- 填写结构化输入
- 查看计划、当前状态、审批、日志和产物

这一层不应该承担的职责是：

- 直接理解 workflow 细节
- 直接消费 Psi 内部事件
- 直接决定权限、审批或工具执行

此前讨论中反复强调的一点是：

`用户应该感知自己在管理任务，而不是在直接操纵 agent 会话。`

所以在体验层，对用户应只暴露：

- `Workspace`
- `Playbook`
- `Task`
- `Approval`
- `Artifact`

不应直接暴露：

- workflow
- routine
- tool call
- psi session

## 四、第二层：应用接口层

这一层是前后端之间的应用命令与查询边界。

它的典型职责包括：

- `Workspace API`
- `Playbook API`
- `Task Command API`
- `Task Query API`
- `Event Stream API`

这一层的定位是：

- 接收 UI 请求
- 返回 UI 所需的结构化数据
- 将产品对象做 DTO 转换

这一层不应该写太多业务决策。

也就是说：

- 它不应该决定“这个动作是否允许”
- 它不应该决定“这个任务进入什么状态”
- 它不应该直接操作工具

它只是把体验层和产品编排层接起来。

## 五、第三层：产品编排层 / 控制面

这一层是 `decacan` 的产品大脑。

此前讨论里，这一层曾经被叫做：

- `Product Domain`
- `Control Plane`
- `decacan-runtime` 的上半层

它的核心职责是：

- 定义产品规则
- 管理任务生命周期
- 解释 `Playbook`
- 生成用户可见计划
- 生成 `Workflow`
- 处理审批
- 管理事件投影
- 登记产物

这层的核心模块包括：

- `Workspace Manager`
- `Playbook Registry`
- `Task Orchestrator`
- `Approval Manager`
- `Artifact Manager`
- `Policy Engine`
- `Event Store`
- `Workflow Compiler`

此前讨论里最关键的一条边界是：

`产品规则必须先于执行逻辑存在。`

也就是说，这一层定义的是：

- 什么是任务
- 什么是正式产物
- 哪些动作可以自动做
- 哪些动作要审批
- 哪些动作直接拒绝

这一层不应该直接做：

- 文件读写
- 命令执行
- 模型推理
- Psi 调用

它关心的是“任务语义”，不是“执行细节”。

## 六、第四层：执行协调层 / 数据面

这一层负责把产品层的任务意图转成真实执行。

此前讨论里，这一层曾被描述为：

- `Execution Coordination`
- `Data Plane`
- `Run Builder + Tool Gateway + Psi Host Adapter`

它的职责包括：

- 将 `Task + Playbook + Policy + Workspace` 编译成一次 `Run`
- 推进 workflow step
- 调用 `Routine`
- 在需要时调用 `Psi`
- 通过 `Tool Gateway` 发起真实动作
- 汇总执行结果
- 将执行事实映射回产品事件

典型模块包括：

- `Run Builder`
- `Run Supervisor`
- `Psi Host Adapter`
- `Execution Mapper`
- `Tool Gateway`
- `Output Writer`
- `Backup Manager`

这里的关键边界是：

- 这一层可以执行
- 但不能定义产品是什么

例如：

- 它可以推进 step
- 它可以调用工具
- 它可以处理备份

但它不应该定义：

- Playbook 在产品里怎么命名
- 任务页面怎么展示
- 什么是主产物，什么是辅助产物

这一层是“把事做出来”的层，而不是“定义产品意义”的层。

## 七、第五层：Psi 内核层

`Psi` 在此前讨论中已经被明确收紧边界：

- 它是项目内可控的执行内核
- 它不是产品壳
- 它不是任务系统
- 它不是审批系统
- 它不是 UI

Psi 负责：

- 接收一次 `PsiInvocation` 输入
- 维护 invocation-local 执行状态
- 在局部语义任务中生成内部计划或分解
- 决定当前 invocation 的下一步局部动作
- 通过宿主受控边界请求工具使用
- 输出稳定 invocation 事件
- 在宿主提供 continuation 后继续执行
- 返回结构化 output candidate 或 invocation result

Psi 不负责：

- task 级计划
- workflow 主循环
- 外层 Run 控制
- task 持久化
- workspace 授权
- approval policy
- approval 对象创建
- artifact 注册
- 产品状态
- 前端 API

此前讨论中，`Psi` 的最小接口被进一步收成：

- `createInvocation(input)`
- `invocation.events()`
- `invocation.resume(continuation)`
- `invocation.result()`

而且有一个硬边界：

`Psi 不应直接碰真实外部动作，只应通过宿主提供的受控工具边界做事。`

同时还补充了一条更硬的边界：

`Psi 不拥有整个 Task / Run 的主循环，它只是在部分 step 中被 runtime 调用。`

## 八、第六层：基础设施层

这一层是最底层的通用资源层。

包括：

- 本地数据库
- 文件系统访问
- 命令执行器
- 模型 provider client
- 配置加载
- 日志系统
- 崩溃恢复钩子

它们不拥有产品语义，只向上提供通用能力。

此前讨论中有一个反复强调的原则：

`基础设施不应该知道 Playbook、Task、Artifact 这些产品对象。`

## 九、用户对象、运行对象、执行对象

为了避免技术分层太抽象，之前的讨论又收敛出三组对象模型：

### 1. 用户对象

- `Workspace`
- `Playbook`
- `Task`
- `Approval`
- `Artifact`

这组对象是产品直接对外的主语言。

### 2. 运行对象

- `Workflow`
- `WorkflowStep`
- `Run`
- `PolicyProfile`
- `ExecutionPlan`
- `TaskEvent`

这组对象是 runtime 的主语言。

### 3. 执行对象

- `Routine`
- `PsiInvocation`
- `ToolRequest`
- `ToolResult`
- `ToolExecutionRecord`
- `OutputWrite`
- `BackupRecord`

这组对象是执行层的主语言。

此前讨论中最关键的一条是：

`前端主要绑定用户对象，runtime 主要持有运行对象，执行层主要处理执行对象。`

## 十、Playbook 这一层

此前架构讨论的一个重大收敛是：

- 不再把产品入口叫 `Template`
- 对内对外统一叫 `Playbook`

原因是：

- `Template` 太像 prompt 模板
- `Playbook` 更强调可复用做事方式、预期产出和内置执行行为

在产品上，用户只看见 `Playbook`，不需要理解 workflow 结构。

`Playbook` 内部则包含：

- Identity
- Input Schema
- Output Contract
- Execution Profile
- Workflow Definition
- Policy Profile

也就是说：

`Playbook = 用户入口 + 输入结构 + 输出合同 + 执行模式 + 隐藏 workflow + 风险策略`

## 十一、模式层：发现模式 / 标准模式

此前讨论中，模式命名经历了多轮收敛，最终定为：

- `发现模式`
- `标准模式`

这两个词描述的是“路径是要被发现，还是已经标准化”，更适合作为产品模式。

关键结论有三条：

1. 模式是产品可见标签
2. 模式不是 workflow 的别名
3. 模式会真实影响 runtime 行为

`标准模式` 的特点：

- workflow 更固定
- 更依赖 routine 与 tool
- `Psi` 比重更低
- 更强调稳定交付
- 更少中断

`发现模式` 的特点：

- workflow 更开放
- `Psi` 比重更高
- 更强调探索与主题发现
- 输出合同更松

此前还确定了：

- 每个 `Playbook` 绑定一个默认模式
- MVP 不做一个 Playbook 在两种模式间自由切换

## 十二、Workflow 这一层

此前讨论中，workflow 被收敛为：

- Playbook 背后的隐藏执行蓝图
- 面向系统，不面向用户

第一版不建议做复杂 DAG 或任意流程图，而建议做：

- 有类型的步骤序列
- 有限分支
- 显式 transition

每个 `WorkflowStep` 至少应有：

- `id`
- `name`
- `type`
- `purpose`
- `input_contract`
- `output_contract`
- `execution_profile`
- `transition`

此前支持的 step type 被收为：

- `deterministic`
- `tool`
- `routine`
- `psi`
- `approval`
- `branch`

此前反复强调的一点是：

`WorkflowStep 要定义工作结构，而不是定义模型怎么想。`

## 十三、Routine 这一层

内部原来使用过 `Skill` 一词，后来在讨论中被明确替换为 `Routine`。

原因是：

- 外部生态里的 “skill” 越来越像可安装能力包
- `decacan` 内部需要的是“例行程序”式的复用单元

所以：

- 外部扩展未来可能叫 `Pack / Extension`
- 内部可复用能力单元统一叫 `Routine`

`Routine` 的定义是：

- workflow 内部可复用执行单元
- 有清楚输入输出契约
- 可由规则、tool 组合、或 Psi 辅助完成
- 不直接对用户暴露

此前讨论里它被解释成：

`Routine = workflow 内部的可复用例行程序。`

## 十四、Tool Gateway 这一层

此前讨论中，`Tool Gateway` 被明确提升为架构关键层，而不是普通工具注册表。

它的真正定位是：

`decacan 中所有真实动作的统一受控入口。`

这意味着：

- `Routine` 不应直接读写文件
- `Psi` 不应直接访问 shell
- workflow step 不应绕过 gateway

此前讨论中，Tool Gateway 至少应承担三件事：

1. 统一动作入口
2. 执行前策略判断
3. 统一结构化结果返回

相关对象包括：

- `ToolDescriptor`
- `ToolRequest`
- `PolicyDecision`
- `ToolResult`
- `ToolExecutionRecord`

其中 `PolicyDecision` 的核心三态被收成：

- `allow`
- `approval_required`
- `deny`

而 `ToolResult` 也应统一为：

- `ok`
- `approval_required`
- `denied`
- `error`

此前架构讨论中，Tool Gateway 是未来接治理层时最重要的兼容边界。

并且补充明确：

- `Psi` 只负责提出 tool request
- runtime 负责把 request 路由到 `Tool Gateway`
- `Tool Gateway` 负责 `allow / approval_required / deny`
- 如果需要审批，由 runtime 创建 `Approval`

## 十五、Task 这一层

此前讨论中，`Task` 被确定为整个产品的中轴对象。

原因是：

- 用户发起的是任务
- runtime 执行的是任务
- 审批、日志、产物都归属于任务

此前对 `Task` 的收敛包括：

- 它不是 prompt
- 它不是 Psi invocation 的别名
- 它是产品级工作实例

Task 结构主要包括五部分：

1. Identity
2. User Intent
3. Runtime Binding
4. Execution State
5. Result Surface

此前收敛出的状态包括：

- `created`
- `planning`
- `running`
- `waiting_approval`
- `completed`
- `failed`
- `cancelled`

最关键的一条是：

`Task 是产品对象，不是内核对象。`

## 十六、Run 这一层

此前讨论中，`Run` 被明确为：

`Task` 下面的一次执行实例，而不是用户主对象。`

之所以需要它，是因为一个任务在未来可能会：

- 中断
- 恢复
- 重试
- 重跑

所以：

- `Task` 表示“这项工作”
- `Run` 表示“这次实际执行”

此前收敛出的 Run 状态包括：

- `initialized`
- `running`
- `paused`
- `completed`
- `failed`
- `cancelled`

而且有一个关键设计原则：

`Psi 最好只感知 invocation 级上下文，不感知 Task 级产品语义。`

因此更准确地说：

- `Task` 是产品工作单
- `Run` 是外层执行实例
- `PsiInvocation` 是 Run 内某一步的智能执行单元

## 十七、Event 这一层

此前讨论中，事件系统被明确分成三层：

1. `Execution Events`
2. `Runtime Events`
3. `Product Events / Task Events`

这样做的原因是：

- 底层执行事实太细，不适合直接给用户看
- 前端需要稳定而简洁的任务状态语言

此前收敛出的 `TaskEvent` 类型包括：

- `task.created`
- `plan.ready`
- `task.running`
- `task.waiting_approval`
- `task.resumed`
- `artifact.ready`
- `task.completed`
- `task.failed`

其中最关键的一条原则是：

`Event 系统的职责，是把执行事实翻译成产品可见状态。`

并补充了一条对 Psi 的要求：

`Psi 原始事件不能直接暴露给前端，必须先经过 runtime 投影。`

## 十八、Artifact 这一层

此前讨论中，`Artifact` 被明确为：

`decacan` 的正式交付对象，不是“顺手写出来的某个文件”。`

这是一个非常关键的产品判断。

此前收敛出的 Artifact 角色包括：

1. 结果承诺
2. 可见交付
3. 追踪锚点
4. 后续工作的输入

其结构包括：

- `id`
- `task_id`
- `workspace_id`
- `playbook_id`
- `kind`
- `type`
- `label`
- `canonical_path`
- `physical_path`
- `status`
- `summary`
- `source_refs`
- `created_at`

此前还收敛出三类 artifact：

- `primary`
- `supporting`
- `backup`

并且明确：

- 文件只有在被 runtime 按输出合同正式登记后，才成为 artifact
- 对 `output/summary.md` 的旧版备份，也可以登记为 backup artifact

同时也补充了：

- `Psi` 或 `Routine` 可以产出 output candidate
- 但只有 runtime 能把 candidate 认定为正式 `Artifact`

## 十九、Approval 这一层

此前讨论中，`Approval` 被明确为：

`正式产品对象，而不是 UI 弹窗。`

这是因为：

- `approval_required` 只是执行层结果
- 真正进入任务生命周期的是 `Approval` 对象

此前收敛出的 Approval 作用包括：

1. 风险边界可见化
2. 暂停点
3. 决策记录
4. 恢复锚点

其状态被收敛为：

- `pending`
- `approved`
- `rejected`
- `resolved`

此前还有一个关键判断：

- `标准模式` 应尽量少触发审批
- `发现模式` 也不意味着应该多审批
- 审批始终只绑定风险动作，而不是绑定探索性推理本身

并补充：

- `approval_required` 是执行层结果
- `Approval` 是产品层对象
- `Psi` 本身不创建 `Approval`

## 二十、首发 Playbook 讨论结果

此前围绕 MVP 产品面还收敛出首发双模式 Playbook：

### 1. 标准模式 Playbook：总结资料

关键定义：

- 输入只支持 Markdown
- 默认扫描整个工作目录，也允许指定子路径
- 默认产物固定为 `output/summary.md`
- 如果已存在则自动备份旧版后覆盖
- 默认全自动，无需审批
- 输出结构固定为：
  - `总览`
  - `主题分组`
  - `关键结论`
  - `信息缺口 / 待确认事项`
  - `建议下一步`
- 正文按内容主题自动归纳，而不是按目录结构组织
- 每个主题下都要标来源 Markdown 文件路径

### 2. 发现模式 Playbook：发现资料主题

关键定位：

- 用于探索目录中材料的主题、关系、空白点
- 不以固定 summary 为唯一目标
- 目的是扩大产品的探索广度

此前对产品策略的关键判断是：

- MVP 同时有两种模式
- 但整体重心应偏向 `标准模式`

## 二十一、对象关系图

此前的静态对象关系，被收成：

```text
Workspace
  ├─ contains -> source files
  └─ hosts -> Tasks

Playbook
  ├─ owns -> Input Schema
  ├─ owns -> Output Contract
  ├─ owns -> Policy Profile
  └─ compiles to -> Workflow

Task
  ├─ belongs to -> Workspace
  ├─ instantiated from -> Playbook
  ├─ bound to -> Workflow snapshot
  ├─ has many -> TaskEvents
  ├─ has many -> Approvals
  ├─ has many -> Artifacts
  └─ has one current -> Run

Run
  ├─ belongs to -> Task
  ├─ executes -> WorkflowSteps
  ├─ emits -> Runtime Events
  ├─ invokes -> Routines
  ├─ invokes -> PsiInvocation
  ├─ invokes -> Tool Gateway
  └─ produces -> execution outputs

Workflow
  ├─ belongs to -> Playbook snapshot
  └─ contains -> WorkflowSteps

WorkflowStep
  ├─ may invoke -> Routine
  ├─ may invoke -> PsiInvocation
  ├─ may invoke -> Tool Gateway
  └─ may create -> Approval requirement

Routine
  ├─ internal reusable procedure
  ├─ may call -> Tool Gateway
  └─ may call -> PsiInvocation

PsiInvocation
  ├─ intelligent executor
  └─ reaches real actions only through -> Tool Gateway

Tool Gateway
  ├─ receives -> ToolRequest
  ├─ evaluates -> PolicyDecision
  ├─ returns -> ToolResult
  └─ records -> ToolExecutionRecord

Approval
  ├─ belongs to -> Task
  ├─ pauses -> Run
  └─ resumes or fails -> Task/Run

Artifact
  ├─ belongs to -> Task
  ├─ written in -> Workspace
  ├─ validated against -> Playbook Output Contract
  └─ surfaced to -> user
```

## 二十二、从 Playbook 到 Artifact 的动态时序

此前动态时序被收成：

```text
用户选择 Workspace
-> 选择 Playbook
-> 填写输入

decacan-app
-> 发起 Task 创建请求

decacan-runtime 控制面
-> 创建 Task
-> 冻结 playbook / policy / workspace snapshot
-> 编译 Workflow
-> 创建 Run
-> 生成短计划
-> 发出 plan.ready

decacan-runtime 数据面
-> 推进 WorkflowStep
-> 调用 Routine / PsiInvocation
-> 通过 Tool Gateway 发起真实动作

Tool Gateway
-> allow / approval_required / deny
-> 执行真实工具或返回中断结果

Runtime
-> 将执行事实映射成 TaskEvent
-> 更新 Task / Run 状态
-> 验证输出合同
-> 注册 Artifact

decacan-app
-> 流式接收事件
-> 展示当前状态
-> 展示审批
-> 展示产物
```

## 二十三、关于未来治理层的架构要求

虽然当前讨论中明确表示：

- 现在不把 `ACES` 作为前提
- 先专注 `decacan` 自己的架构

但此前也收敛出一个重要的未来兼容要求：

`decacan-runtime` 必须保留单一真实动作入口。`

这意味着：

- 所有真实工具动作都走 `Tool Gateway`
- 不要让文件读写散落在 routine 里
- 不要让 Psi 直接接触外部动作

这样未来若引入更强治理服务，不需要推翻产品对象模型。

## 二十四、当前架构讨论的总收口

截至目前，最关键的架构结论是：

1. `decacan` 的产品主对象是 `Playbook`
2. `Task` 是产品中轴对象
3. `Run` 是执行实例，不是用户主对象
4. `Workflow` 是隐藏执行蓝图
5. `Routine` 是内部可复用例行程序
6. `Tool Gateway` 是所有真实动作的统一受控入口
7. `Artifact` 是正式交付对象
8. `Approval` 是正式风险决策对象
9. `Psi` 是被 runtime 在局部 step 中调用的内部窄边界智能执行内核
10. 前端消费产品对象，不消费内核细节
